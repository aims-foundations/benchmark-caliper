# Pipeline Architecture: Cost Optimization

## Overview

The validity analysis pipeline uses a **tiered model routing** architecture on direct Anthropic API calls. The guiding principle: use the cheapest tool that produces equivalent output for each step.

This package mirrors `cli_package/`'s routing 1:1 — the subagent-vs-API-call distinction does not change which model handles which step, so the cost profile is identical.

## Design Principles

### 1. Context Optimization (Primary)
- **Per-page extraction**: each PDF page is processed independently by a Haiku call (via `client.call_parallel` / a local `ThreadPoolExecutor`). Per-call context is ~4K tokens instead of the ~40K an all-in-one approach would require.
- **Minimal per-call context**: each call receives only what it needs. The quote spot-check call gets 3 quotes + the PDF, not the full pipeline state.
- **Script-based composition**: the evaluation prompt is assembled deterministically from YAML files (`scripts/compose_prompt.py`), not generated by an LLM reasoning about what to include.

### 2. Model Routing (Secondary)
- **Haiku** — per-page quote extraction + example/template selection (pattern matching, not reasoning)
- **Sonnet** — consolidation, YAML synthesis, quote spot-check, elicitation, persona simulation, web search (moderate reasoning)
- **Opus** — validity scoring only (deep cross-referencing + judgment)
- **Scripts** — counting, template filling, formatting, JSON/YAML parsing (zero LLM)

### 3. Script Displacement
Four pipeline steps that would otherwise require an LLM now use deterministic scripts:

| Step | Script | What it replaces |
|------|--------|------------------|
| Quote verification (mechanical) | `scripts/verify_quotes.py` | Opus counting quotes and checking IDs |
| Output parsing (JSON/YAML) | `scripts/parse_llm_output.py` | Opus fence-stripping and schema validation |
| Prompt composition | `scripts/compose_prompt.py` | Opus assembling the evaluation prompt |
| Report formatting | `scripts/format_results.py` | Opus reading JSON and formatting a table |

## Cost Estimation

Approximate token usage per pipeline run (29-page PDF):

| Step | Model | Input tokens | Output tokens |
|------|-------|--------------|---------------|
| Per-page extraction | Haiku × 29 | ~4K × 29 = 116K | ~500 × 29 = 14.5K |
| Consolidation | Sonnet × 1 | ~20K | ~15K |
| Benchmark example selection (1b-i) | Haiku × 1 | ~3K | ~50 |
| Benchmark YAML synthesis (1b-ii) | Sonnet × 1 | ~25K | ~15K |
| Quote text spot-check | Sonnet × 1 | ~5K | ~1K |
| Elicitation questions | Sonnet × 1 | ~15K | ~2K |
| Persona simulation* | Sonnet × 1 | ~5K | ~1K |
| Elicitation summary | Sonnet × 1 | ~8K | ~3K |
| Assessment slug (2a) | Haiku × 1 | ~1K | ~50 |
| Base template selection (2b) | Haiku × 1 | ~2K | ~50 |
| Region synthesis (2c) | Sonnet × 1 | ~15K | ~8K |
| Web enrichment | Sonnet × 1 | ~15K | ~8K |
| **Scoring** | **Opus × 1** | **~45K** | **~15K** |
| Scripts | — | 0 | 0 |

*Persona simulation runs only in `--persona` mode.

**Total: 1 Opus call + 7–8 Sonnet calls + ~32 Haiku calls + 7 scripts.**

## Prerequisites

- Python 3.10+ with the dependencies in `requirements.txt`
- An Anthropic API key in `.env` (see `.env.example`)
- `pdftk` or `qpdf` for PDF splitting (optional; `split_pdf.sh` falls back to counting pages via Python regex if neither is installed)
