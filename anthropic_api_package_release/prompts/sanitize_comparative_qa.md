## Validity Dimension Reference

Each question is tagged with one of six validity dimensions. These tags contextualize
what deployment concern the question is probing:

- **IO (Input Ontology):** The set of test-case categories the benchmark covers.
  Misalignment (missing categories, or irrelevant ones) harms content validity.
- **IC (Input Content):** The concrete instances represented by individual datapoints.
  Implementation-level choices can introduce construct-irrelevant variance.
- **IF (Input Form):** The encoding of the input signal — text vs. audio, script,
  infrastructure. Signal-distribution mismatches violate external validity.
- **OO (Output Ontology):** The space of outputs plus the decision rules that score
  them. Cultural misalignment most readily arises in the scoring function's
  interpretive step.
- **OC (Output Content):** Label correctness — whether ground-truth labels match
  regional stakeholder judgment. Annotator-population mismatch violates convergent
  and external validity.
- **OF (Output Form):** The representation of output signals — MCQ vs. open-ended,
  text vs. speech. Mismatch with deployment needs violates external validity.

## Your Task

The user message contains three sections:

1. **Comparator benchmark metadata** — lightweight metadata extracted from the first
   two pages of the comparator benchmark's paper (name, domain, languages, region).
   Use this to understand what the comparator IS, so you can avoid asserting properties
   about it that may be false.

2. **Deployment description** — the expert's description of their real-world AI
   deployment (use case and target population). This is the invariant context that
   both the regional and comparative assessments share. Use it to understand what the
   deployment needs, so you can preserve deployment-side specifics in the questions
   and judge whether a phrase refers to the deployment or to the benchmark.

3. **Original Q&A pairs** — the elicitation questions (with dimension tags) and the
   expert's answers. Produce a cleaned version of each pair.

### What to clean in the QUESTIONS

The questions were generated by the pipeline using the regional benchmark's metadata.
Some questions are purely deployment-focused and need no changes. Others embed specific
properties of the regional benchmark to frame the probe — for example:

- "The benchmark evaluates pronoun resolution and coreference in newswire text, but
  your deployment processes legal contracts" (asserts the benchmark's specific task
  and text genre)
- "The benchmark's toxicity annotations use a 4-point severity scale calibrated
  against US hate speech case law" (asserts the benchmark's specific annotation scheme
  and cultural origin)
- "Your pipeline processes customer support tickets using intent classification and
  slot filling from the benchmark's 12-category taxonomy" (asserts the benchmark's
  specific task set and taxonomy size)

For each question, determine whether it embeds regional-benchmark-specific properties.
If it does:

1. **Identify the deployment concern being probed.** Every question targets a validity
   dimension (tagged as IO, IC, IF, OO, OC, or OF — see definitions above). The
   underlying concern is always about the deployment's needs — what the deployment
   requires, what could go wrong if the benchmark doesn't match. Extract this concern.

2. **Abstract away the benchmark-specific framing.** Replace specific claims about the
   regional benchmark's properties with generic language that probes the same deployment
   concern. Use phrasing like "the benchmark under assessment", "the benchmark's task
   coverage", "the benchmark's scoring dimensions" rather than asserting what those
   specifically are.

   **Critical:** Do not replace a regional-benchmark-specific assertion with a
   comparator-benchmark-specific assertion. The goal is to make the question
   benchmark-agnostic so it probes the deployment need without assuming properties of
   either benchmark. Use the comparator metadata only as a guard — to verify you are
   not accidentally asserting something about the comparator that its metadata
   contradicts.

3. **Preserve the deployment-side specifics.** Keep all details about the deployment
   context, target population, cultural context, and domain requirements. These are the
   invariants. For example, "NCERT-aligned topics, Indian numeral naming conventions
   like lakhs and crores" is deployment context — keep it. "The benchmark evaluates
   pronoun resolution in newswire" is a benchmark property — abstract it.

4. **Keep the dimension tag** [IO], [IC], etc. unchanged.

5. **Preserve the question's probing intent.** The cleaned question should still elicit
   the same answer from the expert. If the original question asks "does X about the
   benchmark match Y about your deployment?", the cleaned version should ask "does the
   benchmark's coverage match Y about your deployment?" without asserting X.

### What to do with the ANSWERS

**Keep answers verbatim.** The expert's answers describe their deployment needs and
expectations. Even where an answer implicitly responds to a benchmark-specific framing
in the question, the substantive content is about the deployment. Do not paraphrase,
summarize, or alter the expert's words.

The one exception: if an answer explicitly names the regional benchmark (e.g., "MILU's
format aligns well"), replace the name with "the benchmark" or "the benchmark under
assessment" as appropriate. Additionally, if an answer implicitly endorses or references
a benchmark-specific property without naming the benchmark — e.g., "yes, the labels are
good" where "the labels" refers to a specific labeling scheme unique to the regional
benchmark — minimally rephrase to make the deployment need explicit rather than leaving
it as an implicit reference to a benchmark property the comparator may not share. This
should be rare — in practice, experts answered in terms of their deployment needs, not
by name or by reference to specific benchmark internals.
